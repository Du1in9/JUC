第 2 章 - 进程与线程

2.1 进程与线程

2.1.1 进程

  • 程序由指令和数据组成，但这些指令要运行，数据要读写，就必须将指令加载至 CPU，数据加载至内存。 在指令运行过程中还需要用到磁盘、网络等设备。进程就是用来加载指令、管理内存、管理 IO 的。

  • 当一个程序被运行，从磁盘加载这个程序的代码至内存，这时就开启了一个进程。

  • 进程就可以视为程序的一个实例。大部分程序可以同时运行多个实例进程（例如记事本、画图、浏览器等）， 也有的程序只能启动一个实例进程（例如网易云音乐、360 安全卫士等）。

2.1.2 线程

  • 一个进程之内可以分为一到多个线程。

  • 一个线程就是一个指令流，将指令流中的一条条指令以一定的顺序交给 CPU 执行。

  • Java 中，线程作为最小调度单位，进程作为资源分配的最小单位。 在 windows 中进程是不活动的，只是作为线程的容器。

2.1.3 二者对比

  • 进程基本上相互独立的，而线程存在于进程内，是进程的一个子集

  • 进程拥有共享的资源，如内存空间等，供其内部的线程共享

  • 进程间通信较为复杂

    • 同一台计算机的进程通信称为 IPC（Inter-process communication）

    • 不同计算机之间的进程通信，需要通过网络，并遵守共同的协议，例如 HTTP

  • 线程通信相对简单，因为它们共享进程内的内存，一个例子是多个线程可以访问同一个共享变量

  • 线程更轻量，线程上下文切换成本一般上要比进程上下文切换低

2.2 并行与并发

2.2.1 并行

  • 单核 cpu 下，线程实际还是 串行执行 的。操作系统中有一个组件叫做任务调度器， 将 cpu 的时间片分给不同的程序使用，只是由于 cpu 在线程间（时间片很短）的切换非常快，人类感觉是 同时运行的 。

  • 总结为一句话就是： 微观串行，宏观并行 。一般会将这种 线程轮流使用 CPU 的做法称为并发， concurrent。

2.2.2 并发

  • 多核 cpu下，每个 核（core） 都可以调度运行线程，这时候线程可以是并行的。

  • 并发（concurrent）是同一时间应对（dealing with）多件事情的能力 并行（parallel）是同一时间动手做（doing）多件事情的能力

  • 家庭主妇做饭、打扫卫生、给孩子喂奶，她一个人轮流交替做这多件事，这时就是并发 家庭主妇雇了保姆一起做事，这时既有并发，也有并行（这时产生竞争，例如锅只有一口，一个人用锅时，另一个人就得等待） 雇了3个保姆，一个专做饭、一个专打扫卫生、一个专喂奶，互不干扰，这时是并行

2.3 应用：异步调用

多线程可以让方法执行变为异步的（即不要巴巴干等着）。

比如说读取磁盘文件时，假设读取操作花费了 5 秒钟，如果没有线程调度机制，这 5 秒 cpu 什么都做不了，其它代码都得暂停...

  • 需要等待结果返回，才能继续运行就是同步

2.4 应用：提升效率

  1. 单核 cpu 下，多线程不能实际提高程序运行效率， 只是为了能够在不同的任务之间切换，不同线程轮流使用cpu ，不至于一个线程总占用 cpu，别的线程没法干活。

  2. 多核 cpu 可以并行跑多个线程，但能否提高程序运行效率还是要分情况的有些任务， 经过精心设计，将任务拆分，并行执行，当然可以提高程序的运行效率。 但不是所有计算任务都能拆分（阿姆达尔定律）也不是所有任务都需要拆分，任务的目的如果不同，谈拆分和效率没啥意义。

  3. IO 操作不占用 cpu，只是我们一般拷贝文件使用的是【阻塞 IO】，这时相当于线程虽然不用 cpu， 但需要一直等待 IO 结束，没能充分利用线程。所以才有后面的【非阻塞 IO】和【异步 IO】优化。

3.4 线程的运行原理

3.4.1 栈与栈顿

JVM（Java Virtual Machine）由堆、栈、方法区组成，其中栈就是给线程用。每个线程启动后，会为其分配一块栈内存。

  • 每个栈由多个 栈帧（Frame）组成，对应着每次方法调用时所占用的内存

  • 每个线程只能有一个活动栈帧，对应着当前正在执行的那个方法

xxxxxxxxxx
public static void main(String[] args) {    // TestFrames:5
    method1(10);
}


private static void method1(int x) {        // TestFrames:10
    int y = x + 1;
    System.out.println(method2());
}
private static Object method2() {           // TestFrames:15
    return new Object();
}

3.4.2 线程上下文切换（ThreadContextSwitch）

因为以下一些原因导致 cpu 不再执行当前的线程，转而执行另一个线程的代码：

  • 线程的 cpu 时间片用完

  • 垃圾回收

  • 有更高优先级的线程需要运行

  • 线程自己调用了 sleep、yield、wait、join、park、synchronized、lock 等方法

当 Context Switch 发生时，需要由操作系统保存当前线程的状态，并恢复另一个线程的状态。

Java 中程序计数器（Program Counter Register）的作用是记住下一条 JVM 指令的执行地址，是线程私有的。

  • 状态包括程序计数器、虚拟机栈中每个栈帧的信息，如局部变量、操作数栈、返回地址等

  • Context Switch 频繁发生会影响性能

3.11 五种状态 - 操作系统层面

  1. 【初始状态】仅是在语言层面创建了线程对象，还未与操作系统线程关联。

  2. 【可运行状态】（就绪状态）指该线程已经被创建（与操作系统线程关联），可以由 CPU 调度执行。

  3. 【运行状态】获取了 CPU 时间片运行中的状态。当CPU时间片用完，进入【可运行状态】，线程上下文切换。

  4. 【阻塞状态】如果调用了阻塞 API，如 BIO 读写文件，这时线程实际不会用到 CPU，进入【阻塞状态】，线程上下文切换。

  5. 【终止状态】表示线程已经执行完毕，生命周期已经结束，不会再转换为其它状态。

3.12 六种状态 - Java API 层面

  • NEW 线程刚被创建，但是还没有调用 start() 方法。

  • RUNNABLE 当调用了 start() 方法之后， 涵盖了【可运行状态】、【运行状态】和【阻塞状态】。

  • BLOCKED ， WAITING ， TIMED_WAITING 都是对【阻塞状态】的细分。

  • TERMINATED 当线程代码运行结束。

4.1 共享带来的问题

4.1.1 Java 的体现

xxxxxxxxxx

4.1.2 问题分析

  • 一个程序运行多个线程本身是没有问题的，问题出在多个线程访问共享资源。

  • 一段代码块内如果存在对共享资源的多线程读写操作，称这段代码块为 临界区。

  • 多个线程在临界区内执行，由于代码的执行序列不同而导致结果无法预测，称之为发生了 竞态条件。

  • 为了避免临界区的竞态条件发生，有多种手段可以达到目的。

    • 阻塞式的解决方案：synchronized，Lock

    • 非阻塞式的解决方案：原子变量

4.4.2 原理之 Monitor（锁）

Monitor 被翻译为监视器或管程，每个 Java 对象都可以关联一个 Monitor 对象。

如果使用 synchronized 给对象上锁（重量级）之后，该对象头的 MarkWord 中就被设置指向 Momitor 对象的指针。

  1. 当 Thread-2 执行 synchronized 就会将 Monitor 的所有者 Owner 置为 Thread-2，Monitor 中只能有一个 Owner

  2. 在 Thread-2 上锁的过程中，如果 Thread-3，Thread-4，Thread-5 也来执行 synchronized，就会进入 EntryList 阻塞

  3. 当 Thread-2 执行完同步代码块的内容后，唤醒 EntryList 中等待的线程来竞争锁，竞争是非公平的

  4. 图中 WaitSet 中的 Thread-0，Thread-1 是之前获得过锁，但条件不满足进入 WAITING 状态的线程

4.5 wait / notify

4.5.1 wait / notify 原理

  1. Owner 线程发现条件不满足，调用 wait 方法，即可进入 WaitSet 变为 WAITING 状态

  2. BLOCKED 和 WAITING 的线程都处于阻塞状态，不占用 CPU 时间片

  3. BLOCKED 线程会在 Owner 线程释放锁时唤醒

  4. WAITING 线程会在 Owner 线程调用 notify 时唤醒，但不立刻获得锁，而是进入 EntryList 重新竞争

4.5.2 API 介绍

  • obj.wait() 让进入 object 监视器的线程到 waitSet 等待

  • obj.notify() 在 object 上正在 waitSet 等待的线程中挑一个唤醒

  • obj.notifyAll() 让 object 上正在 waitSet 等待的线程全部唤醒

xxxxxxxxxx
private void task() {
    synchronized (obj) {
        try {
            obj.wait();
        } catch (InterruptedException e) {
            e.printStackTrace();
        }
    }
}


new Thread(this::task).start();
new Thread(this::task).start();


synchronized (obj) {
    // obj.notify(); 
    // obj.notifyAll();
}

4.6 wait / notify 使用

4.6.1 sleep(long n) 和 wait(long n) 的区别

  1. sleep 是 Thread 方法，而 wait 是 Object 的方法

  2. sleep 不需要强制和 synchronized 配合使用，但 wait 需要和 synchronized 一起用

  3. sleep 在睡眠的同时，不会释放对象锁，但 wait 在等待的时候会释放对象锁

4.6.2 模式之保护性暂停

这个模式主要用于在多线程环境中处理某个线程需要等待某种条件满足才能继续执行的情况。

  • 一个线程需要某个条件才能继续执行，如果条件不满足，它会暂停并等待（wait）。

  • 一旦条件满足，其他线程会发出通知（notify），唤醒等待的线程，让它继续执行。

4.6.3 模式之生产者消费者

这个模式主要用于协调两个或多个线程之间的工作流程，关注如何安全地在多个线程之间共享数据。

  • 其中一个线程（生产者）生成数据并将其放入一个共享缓冲区。

  • 而另一个线程（消费者）从共享缓冲区中取出数据并进行处理。

第 5 章 - 共享模型：内存

上一章讲的 Monitor 主要关注的是访问共享变量时，保证临界区代码的原子性。

这一章讲共享变量在多线程间的可见性问题，与多条指令执行时的有序性问题。

5.1 Java 内存模型

JMM（Java Memory Model），定义了主存、工作内存抽象概念，底层对应着 CPU 寄存器、缓存、硬件内存、CPU 指令优化等。

  • 原子性 - 保证指令不会受到线程上下文切换的影响

  • 可见性 - 保证指令不会受 cpu 缓存的影响

  • 有序性 - 保证指令不会受 cpu 指令并行优化的影响

5.2 可见性

5.2.1 问题与解决

xxxxxxxxxx
static boolean run = true;


public static void main(String[] args) throws InterruptedException {
    Thread t = new Thread(() -> while(run) {...});
    t.start();
    
    run = false;        // 线程t不会如预想的停下来
}

问题：main 线程对 run 变量的修改对于 t 线程不可见，导致了 t 线程无法停止

解决：volatile（易变关键字）可以用来修饰成员变量和静态成员变量，避免线程从工作缓存中查找值，而是直接操作主存。

xxxxxxxxxx
volatile static boolean run = true;

但 volatile 不能保证原子性，它仅用在一个写线程，多个读线程的情况。

5.2.2 设计模式：两阶段终止

在 Java 线程设计中，两阶段终止模式是一种优雅地停止线程的方法，它允许线程在终止前完成当前的任务。

这种模式特别适用于那些需要在停止前进行清理操作或者需要确保任务能够完整执行的线程。

  1. 中断阶段：在这个阶段，线程可以检查中断标志，并决定是否继续执行当前任务。

  2. 终止阶段：在这个阶段，线程可以会检查一个明确的终止标志，然后安全地退出。

xxxxxxxxxx
public class TwoPhaseTermination {
    public static void main(String[] args) throws InterruptedException {
        WorkerThread worker = new WorkerThread();
        worker.start();
        Thread.sleep(3000);
        worker.stop();                  // 优雅停止线程
    }
}

class WorkerThread extends Thread {
    private volatile boolean stop = false;


    @Override

5.2.5 设计模式：Balking

Balking（犹豫）模式是一种用于处理状态变更的设计模式，特别适合于那些需要避免某些操作在 不合适的状态 下执行的场景。

它的核心思想是：如果对象的状态不适合执行某项操作，操作将被拒绝，对象将保持原有状态。使用场景如资源管理、任务调度等。

});

    balkingData.changeData("Another data");
    balkingData.saveData();
});


t1.start();
t2.start();

public class BalkingData {

    private boolean changed;

        this.changed = false;
    }


    public synchronized void changeData(String newData) {   // 1. 更改数据，但如果数据已经更改，它会拒绝再次更改

        this.changed = true;

    }

5.3.3 volatile 原理

  1. 保证可见性：

    • 读屏障（lfence）保证在该屏障之后，对共享变量的读取，加载的是主存数据

    • 写屏障（sfence）保证在该屏障之前，对共享变量的改动，都同步到主存当中

  2. 保证有序性：

    • 读屏障（lfence）会确保指令重排序时，不会将读屏障之后的代码排在之前

    • 写屏障（sfence）会确保指令重排序时，不会将写屏障之前的代码排在之后

xxxxxxxxxx
public void actor1(I_Result r) {
    if(ready) r.val = num * 2;              // 读屏障
    else r.val = 1;
}
xxxxxxxxxx
public void actor2(I_Result r) {
    num = 2;
    ready = true;                           // 写屏障
}

注意：volatile 仅仅保证了可见性、有序性，但不能保证原子性，即不能解决指令交错问题。

6.2 CAS

6.2.1 CAS 与 volatile

AtomicInteger 内部并没有用锁来保护共享变量的线程安全，其中的关键是 compareAndSet（CAS），它是原子操作。

xxxxxxxxxx
public void withdraw(Integer amount) {
    while (true) {
        int prev = balance.get();
        int next = prev - amount;
        if (balance.compareAndSet(prev, next)) {
            break;
        }
    }
}

注意：CAS 必须借助 volatile 才能读取到共享变量的最新值，来实现【比较并交换】的效果。

6.2.2 效率分析

  • 无锁情况下，始终在高速运行，而 synchronized 会在没有获得锁的时候，会进入阻塞状态，导致上下文切换。

    线程就好像高速跑道上的赛车，一旦发生上下文切换，就要减速、熄火。等被唤醒又得重新打火、启动、加速等。

  • 但如果没有额外 CPU 的支持，线程就不能高速运行。由于线程没有分到时间片，仍会进入可运行状态，导致上下文切换。

6.2.3 CAS 特点

结合 CAS 和 volatile 可以实现无锁并发，适用于线程数少、多核 CPU 的场景下。

  • CAS 基于 乐观锁 的思想：不怕别的线程来修改共享变量，就算改了也没关系，我吃亏点再重试呗。

  • synchronized 基于 悲观锁 的思想：防着别的线程来修改共享变量，我上了锁你们都别想改，我改完了才解开锁。

  • CAS 体现的是无锁并发、无阻塞并发：

    • 因为没有使用 synchronized，所以线程不会陷入阻塞，这是效率提升的因素之一。

    • 但如果竞争激烈，可以想到重试必然频繁发生，反而效率会受影响。

8.1.2.1 线程池状态

ThreadPoolExecutor 使用 int 的高 3 位来表示线程池状态，低 29 位表示线程数量：

从数字上比较，TERMINATED > TIDYING > STOP > SHUTDOWN > RUNNING。

这些信息存储在一个原子变量 ctl 中，目的是将线程池状态与线程个数合二为一，这样就可以用一次 CAS 原子操作进行赋值。

xxxxxxxxxx

// rs 为高 3 位代表线程池状态， wc 为低 29 位代表线程个数，ctl 是合并它们

8.1.2.2 构造方法

  1. corePoolSize：核心线程数目

  2. maximumPoolSize：最大线程数目 = 核心线程数目 + 非核心线程数目

  3. keepAliveTime：生存时间，生存时间内没有新任务，此线程资源会释放

  4. unit：生存时间单位，如秒、毫秒等

  5. workQueue：阻塞队列，当没有空闲核心线程时，新来任务会加入到此队列排队

  6. threadFactory：线程工厂，可以定制线程对象的创建，例如设置线程名字

  7. handler：拒绝策略，当所有线程都在忙，阻塞队列也放满时，会触发拒绝策略

  • jdk 拒绝策略 RejectedExecutionHandler 提供了 4 种实现，其它常见框架也提供了实现：

    1. AbortPolicy 让调用者抛出 RejectedExecutionException 异常，这是默认策略

    2. CallerRunsPolicy 让调用者运行任务

    3. DiscardPolicy 放弃本次任务

    4. DiscardOldestPolicy 放弃队列中最早的任务，本任务取而代之

    • Dubbo 的实现，在抛出 RejectedExecutionException 异常之前会记录日志，并 dump 线程栈信息，方便定位问题

    • Netty 的实现，是创建一个新线程来执行任务

    • ActiveMQ 的实现，带超时等待（60s）尝试放入队列，类似我们之前自定义的拒绝策略

    • PinPoint 的实现，它使用了一个拒绝策略链，会逐一尝试策略链中每种拒绝策略

  • 当高峰过去后，救急线程如果一段时间没有任务做，需要结束节省资源，这个时间由 keepAliveTime 和 unit 来控制

8.1.2.12 Tomcat 线程池

  1. Tomcat 在哪里用到了线程池呢？

    • Acceptor 只负责【接收新的 socket 连接】

                throw rx;
            }
        }
    }

8.1.3 Fork / Join

  1. 概念

    • Fork / Join 是 jdk 1.7 加入的新的线程池实现，它体现的是一种分治思想，适用于能够进行任务拆分的 CPU 密集型运算。

      • 所谓的任务拆分，是将一个大任务拆分为算法上相同的小任务，直至不能拆分可以直接求解。

      • 跟递归相关的一些计算，如归并排序、斐波那契数列、都可以用分治思想进行求解。

    • Fork / Join 在分治的基础上加入了多线程，可以把每个任务的分解和合并交给不同的线程来完成，进一步提升了运算效率。

    • Fork / Join 默认会创建与 CPU 核心数大小相同的线程池。

  2. 使用

    提交给 Fork / Join 线程池的任务需要继承 RecursiveTask（有返回值）或 RecursiveAction（没有返回值）

    • 例如下面定义了一个对 1~n 之间的整数求和的任务

    xxxxxxxxxx
    class AddTask1 extends RecursiveTask<Integer> {
        int n;
        
        public AddTask1(int n) { this.n = n; }
        
        @Override
        public String toString() { return "{" + n + '}'; }
        
        @Override
        protected Integer compute() {
            if (n == 1) return n;
            
            AddTask1 t1 = new AddTask1(n - 1);
            t1.fork();
            return n + t1.join();
        }
    }

    • 然后提交给 ForkJoinPool 来执行：

    xxxxxxxxxx
    public static void main(String[] args) {
        ForkJoinPool pool = new ForkJoinPool(4);
        System.out.println(pool.invoke(new AddTask1(5)));       // 15
    }

8.2.3 读写锁

8.2.3.1 ReentrantReadWriteLock

ReentrantReadWriteLock 是一种读写锁，它允许多个线程同时读取共享资源，但写操作会独占资源。主要特点是：

  • 读锁：多个线程可以同时获得读锁，只要没有线程持有写锁。

  • 写锁：写锁是独占的，获取写锁的线程会阻止其他线程获取读锁或写锁。

  1. 使用示例：

xxxxxxxxxx
private final ReentrantReadWriteLock rwLock = new ReentrantReadWriteLock();


public void readData() {
    rwLock.readLock().lock();
    try {
        System.out.println("Reading...");
    } finally {
        rwLock.readLock().unlock();
    }
}


public void writeData() {
    rwLock.writeLock().lock();
    try {
        System.out.println("Writing...");
    } finally {
        rwLock.writeLock().unlock();
    }
}
xxxxxxxxxx
public static void main(String[] args) {

    new Thread(() -> example.readData()).start();       // 3. 读操作
}

  1. 注意事项：

    • 避免死锁：不要同时持有多个锁，确保获取锁的顺序一致。

    • 性能考量：大量的读操作和少量的写操作情况下，能显著提高性能。但在写操作频繁的场景下，可能会降低性能。

    • 锁的粒度：确保锁定的粒度合理，避免过多锁的竞争和潜在的性能问题。

  1. 缓存应用

    • 在缓存实现中，ReentrantReadWriteLock 适用于读操作远多于写操作的场景，如缓存读取远多于更新的情况。

    • 读操作可以并发进行，而写操作可以安全地更新缓存内容。

8.2.3.2 StampedLock

StampedLock 是 Java 8 引入的高级锁，它提供了比 ReentrantReadWriteLock 更灵活的锁机制，特别适用于高并发的读写场景。

StampedLock 允许获取乐观读锁，支持无阻塞的读操作和改进的写锁策略。

  1. 原理：

    • 悲观读锁：类似于 ReentrantReadWriteLock 的读锁，持有锁期间其他线程不能修改数据。

    • 乐观读锁：允许线程在不获取锁的情况下读取数据，但在读取过程中可能需要检查数据是否被修改过。

    • 写锁：独占的锁，写操作会阻塞其他的读锁和写锁。

  2. 使用示例：

xxxxxxxxxx
private final StampedLock lock = new StampedLock();


public void readValue() {
    long stamp = lock.readLock();
    try {
        System.out.println("Reading...");
    } finally {
        lock.unlockRead(stamp);
    }
}


public void optimisticReadValue() {
    long stamp = lock.tryOptimisticRead();


    if (!lock.validate(stamp)) {
        stamp = lock.readLock();
        try {
            System.out.println("Reading...");
        } finally {
            lock.unlockRead(stamp);
        }
    }
}


public void writeValue(int newValue) {
    long stamp = lock.writeLock();
    try {
        System.out.println("Writing...");
    } finally {
        lock.unlockWrite(stamp);
    }
}

8.2.7 线程安全集合类

  1. 遗留的线程安全集合：

    • Vector：双端队列的实现，支持同步操作。由于其所有方法都是同步的，可能会影响性能。

    • Stack：继承自 Vector，实现了栈的数据结构。由于 Vector 的同步机制，Stack 也具有相同的同步特性。

    • Hashtable：提供键值对存储的线程安全的 Map，与 ConcurrentHashMap 相比，性能较差，因为它的所有方法都是同步的。

  2. 使用 Collections 装饰的集合：

    这些装饰类是线程安全的，但由于所有操作都加了同步锁，可能会影响性能。

    • Collections.synchroizedList：将普通 List 通过包装成线程安全的 List。注意，对装饰后的 List 的操作需手动同步迭代器。

    • Collections.synchronizedMap：将普通 Map 包装成线程安全的 Map。访问集合时需要同步整个 Map，特别是迭代操作。

    • Collections.synchronizedSet：将普通 Set 包装成线程安全的 Set。类似于其他包装集合，操作时需要手动同步迭代器。

  3. 使用 java.util.concurrent 包的集合：

    • ConcurrentHashMap：分段锁机制提高了并发性，不会对整个 Map 锁定，而是划分为多个段，锁定操作仅限于特定段。

    • BlockingQueue 接口实现：提供了线程安全的队列操作，支持阻塞的队列操作，如 put 和 take 方法。

    • ConcurrentLinkedQueue：基于非阻塞算法的线程安全队列，适合高并发的场景。

    • CopyOnWriteArrayList：在每次写操作时复制整个数组，因此读取操作不会受到影响。适用于读操作远多于写操作的场景。

    • CopyOnWriteArraySet：类似于 CopyOnWriteArrayList，用于需要线程安全的 Set，适合读多写少的场景。

    • ConcurrentSkipListMap 和 ConcurrentSkipListSet：基于跳表的数据结构，实现了线程安全，支持有序操作和高并发。

8.2.7.4 CopyOnWriteArrayList

CopyOnWriteArrayList 是一个线程安全的变种 ArrayList。它通过在每次修改时复制整个底层数组来确保线程安全。

这种实现策略使得在写操作（增删改）时，必须复制整个数组，从而牺牲了性能，但提供了高效的读操作。适用于读多写少的场景。

xxxxxxxxxx
List<String> list = new CopyOnWriteArrayList<>();


list.add("apple");
list.add("banana");
list.add("cherry");


for (String fruit : list) {
    System.out.println(fruit);                              // apple banana cherry
}


list.add("date");
System.out.println(list);                                   // [apple, banana, cherry, date]
